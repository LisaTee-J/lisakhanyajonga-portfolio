### Reflection 1: Coding in Markdown 

**Situation**  
As part of my PRP370 Project Presentation module at CPUT, I was required to create a complete digital portfolio using GitHub Flavoured Markdown (GFM). Before this assessment, I had only used word processors like Microsoft Word and had no experience with Markdown syntax.

**Task**  
I needed to learn Markdown from scratch and use it to structure my entire professional portfolio, including my CV, video embedding, and all reflections. The portfolio had to be clean, readable, and professionally formatted without using traditional word processing tools.

**Action**  
I started by reading the official GitHub Markdown documentation and practicing basic syntax on Youtube. I learned to use:
- `#` for headings and `##` for subheadings to create a clear hierarchy
- `-` for bullet points in my skills and experience sections
- `**bold**` and `*italic*` for emphasis
- `[link text](URL)` for hyperlinks
- `---` for horizontal rules to separate sections
- Backticks (`` ` ``) for code formatting
- HTML `<video>` tags for embedding my mock interview video

I created a table of contents using anchor links, organized my CV into logical sections, and used proper spacing to improve readability. I also learned to use GitHub's preview feature to check my formatting before committing changes.

**Result**  
I successfully built a fully formatted, professional-looking portfolio using only plain text Markdown. The final result is:
- Easy to read on both GitHub and GitHub Pages
- Faster to edit than a Word document
- Version-controlled through Git
- Responsive and works on mobile devices
- Professionally formatted with clear section separation

This experience taught me a valuable developer skill. I now understand why Markdown is the standard for documentation on platforms like GitHub, Stack Overflow, and Reddit. I can confidently use Markdown for future projects, README files, and technical documentation.

### Reflection 3: Using GitHub Pages (STAR Method)

**Situation**  
After creating my Markdown portfolio in a GitHub repository, I needed to deploy it to a live, publicly accessible website. I had never used GitHub Pages before and was unsure how static site hosting worked.

**Task**  
The goal was to publish my portfolio so that:
- It could be accessed via a unique URL (e.g., `https://username.github.io/repository-name/`)
- The URL could be submitted on Blackboard for assessment
- The site would automatically update when I pushed changes to my repository
- The deployment process had to be completed without additional cost or external hosting services

**Action**  
I followed these steps to deploy my portfolio:

1. Navigated to my repository on GitHub.com
2. Clicked on the **"Settings"** tab (near the top right of the page)
3. Scrolled down to find **"Pages"** in the left sidebar under "Code and automation"
4. Under the "Branch" section, I changed the source from "None" to **"main"**
5. Left the folder as **"/ (root)"** since my `README.md` was in the main directory
6. Clicked the **"Save"** button
7. Waited approximately 1-2 minutes for GitHub to build and deploy my site
8. Refreshed the page until I saw the message: *"Your site is live at `https://username.github.io/repo-name/`"*
9. Clicked the link to verify my portfolio was displaying correctly
10. Tested the link on both desktop and mobile devices

**Result**  
My portfolio is now successfully deployed and live on the internet at a professional, shareable URL. Key achievements:
- **Free hosting:** GitHub Pages provided this at no cost
- **Automatic updates:** Any future changes I push to my `README.md` automatically update the live site
- **Custom URL:** I can optionally add a custom domain later
- **Professional presentation:** The portfolio looks identical to my local Markdown preview
